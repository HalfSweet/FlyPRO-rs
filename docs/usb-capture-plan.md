# USB 抓包计划

本计划用于把静态反汇编事实逐步提升为可回放的真机协议证据。默认使用 Windows 官方
FlyPRO II 软件和 USBPcap/Wireshark；Linux `usbmon` 可用于后续交叉验证，但不能替代官方
上位机产生的基线。任何未知命令实验都不得使用生产芯片、唯一设备或不可恢复的编程器。

## 1. 静态分析能得到什么

当前上位机、驱动和资产已经静态确认：

- 正常模式设备接口 GUID、`VID_5346&PID_5109` 和 Windows 打开流程；
- Pipe 地址及方向：`0x02/0x03` OUT，`0x82/0x83/0x84/0x85` IN；
- 64 字节控制块、常见 `0x800` 字节应用分块和各 Pipe 调用的精确长度/超时；
- `0x0008` 算法身份校验、`0x0087` 算法分块、`0x008A` 器件参数下发的字段与阶段；
- 一批尚未命名的命令常量、调用地址和静态事务形状；
- `.alg`、`SP20.dev`、`.cfg` 的已确认字段以及算法选择关系。

继续反汇编有机会还原更多命令字段和 2048 字节参数构造器，但静态证据不能可靠回答 UI
操作与未知命令的对应关系，也不能证明设备实际返回值的含义。

以下内容必须由真机补齐或验证：

- 设备、配置、接口、alternate setting、端点传输类型和 `wMaxPacketSize`；
- 连接、身份、读 ID、读、查空、擦除、编程、校验与命令 ID 的一一对应；
- 地址、长度、区域、电压、时钟等动态字段和跨阶段关联规则；
- `0x82` 状态位、`0x84/0x85` 返回块、错误码、超时和取消后的真实语义；
- 2048 字节器件参数映像的完整生成规则；
- 不同编程器型号、固件以及正常/升级模式的 VID/PID 和端点差异。

## 2. 每份抓包必须附带的信息

每个用例独立重启抓包，并保存完整 `.pcapng`，不要只截取看起来有用的帧。同时记录：

- 操作系统版本、官方软件版本及主程序 SHA-256；
- 编程器型号、序列号、固件版本和 USB 描述符 JSON；
- 目标芯片完整型号、封装、适配座、供电方式、电压和时钟；
- `SP20.dev` 记录偏移，`.alg`/`.cfg` 文件名和 SHA-256；
- 开始/结束时间、唯一用例 ID、准确 UI 操作和预期结果；
- 输入数据文件 SHA-256；读取类操作还要保存软件导出的原始结果文件；
- 成功/失败截图和官方软件日志。

首次接入时先运行：

```bash
flypro usb list --json
flypro usb inspect --index 0 --json
```

这两条命令不 claim 接口、不发端点传输，可先关闭官方软件执行。描述符 JSON 与对应抓包
使用同一个用例 ID 命名。

## 3. P0：非破坏性基线

建议固定一颗 `W25Q128BV` 测试芯片和现有 `w25q128.alg` 基线，依次采集：

| 用例 | 唯一变量 | 要回答的问题 |
|---|---|---|
| `DISC-01` | 插入编程器并让官方软件自动连接 | 枚举、打开、初始化和初始身份事务 |
| `DISC-02` | 拔插后重新连接 | 会话重建、设备地址变化和重试行为 |
| `INFO-01` | 读取编程器型号/序列号/固件版本 | 身份命令、响应长度和字段 |
| `ALG-01` | 新启动官方软件后选择 W25Q128BV | 新会话算法下载与 2048 字节参数映像 |
| `ALG-02` | 同一会话再次执行同一操作 | 算法复用路径和 `0x0008` 响应 |
| `ALG-03` | 切换到另一算法再切回 | 算法不匹配、重新下载和参数差异 |
| `ID-01` | 只执行“读取芯片 ID” | 读 ID 命令、响应和状态语义 |
| `READ-01` | 从地址 0 读取 256 字节 | 最小完整读取事务 |
| `READ-02` | 仅把地址改为 `0x100` | 定位命令中的地址字段 |
| `READ-03` | 地址不变，长度依次为 1/16/64/256/2048/2049 | 定位长度字段和应用分块边界 |
| `BLANK-01` | 对已知全 `0xff` 样片查空 | 查空成功路径 |
| `BLANK-02` | 仅改变一个字节后再次查空 | 查空失败位置和错误返回 |

如果选择器件本身会给目标上电，`ALG-*` 必须接入限流电源和正确方向的测试座；不能把“仅在
UI 选择”默认当成无电气动作。

## 4. P1：可牺牲样片上的写操作

只有在 P0 解析器能稳定重放、且样片内容已经备份后才执行：

1. `ERASE-01`：对非空测试样片擦除，随后用官方软件读取并查空；
2. `PROG-01..04`：分别写入全 `00`、全 `ff`、递增字节和固定伪随机模式；
3. 对同一模式只改变地址，再只改变长度，覆盖 2048 字节分块边界；
4. `VERIFY-01`：校验相同文件成功；
5. `VERIFY-02`：只修改输入文件一个字节，获得可定位的校验失败；
6. 配置区/OTP/保护位另开批次，必须先读取备份并确认可恢复性。

不要在未知执行状态后自动重发擦除或编程。超时、短传输或拔线都应保留为单独失败用例，
由人工确认器件状态后再继续。

## 5. P2：故障和兼容矩阵

协议主路径稳定后再补充：

- 无芯片、芯片反接、接触不良、ID 不匹配和目标外部供电；
- 只读操作中的取消、超时和安全拔线；
- SP10/SP20 不同型号、至少两个固件版本；
- Windows 10/11 的官方基线，以及 Rust 实现在 Linux/macOS/Windows 的相同读操作；
- 正常模式与 bootloader/升级模式；升级抓包只在具备恢复手段时执行；
- 脱机项目下载最后处理，并用只改变一个项目字段的二进制差分配合抓包。

## 6. 证据闭环标准

一个操作只有同时满足以下条件才进入 Rust 公共 API：

1. 至少一组成功和一组可解释失败抓包；
2. 单变量对照能定位命令、字段、阶段、精确长度和超时；
3. 原始事务已转成脱离硬件可运行的 golden test；
4. 写操作明确标注破坏性、取消规则和“未知执行状态”行为；
5. 在真机上对官方软件与 Rust 实现的最终数据进行逐字节比对。
