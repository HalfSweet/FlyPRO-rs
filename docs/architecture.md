# 架构设计

本文描述 FlyPRO-rs 的高层结构、模块职责和依赖原则。协议字段、资产布局与事务顺序见
[实现细节](implementation.md)。

## 设计目标

- 为硕飞 SP10/SP20 烧录器提供跨平台的 Rust 工具；
- 将 FlyPro 软件发布资产的解析与 USB 操作从命令行界面中分离；
- 只把已有证据支持的行为表达为稳定类型，未知字段保持原始形式；
- 对读取和写入流程使用同一套可测试的会话状态机；
- 在传输失败或设备状态不明时停止操作，避免隐式重放写命令。

## Workspace 边界

```text
flypro-cli
    └── flypro-core
            ├── assets
            ├── protocol
            ├── parameters
            ├── session
            ├── operations
            ├── transport
            ├── usb
            └── usb_transport
```

### `flypro-core`

核心库是资产格式和设备协议的唯一实现位置：

- `assets` 解析器件数据库、封装映射、算法和配置；
- `protocol` 定义命令块、响应和已确认的状态谓词；
- `parameters` 构造并校验器件参数镜像；
- `session` 准备算法并管理单次连接内的算法状态；
- `operations` 编排查空、读写、配置和擦除等操作；
- `transport` 定义与平台无关的精确传输接口；
- `usb` 负责设备发现和描述符检查；
- `usb_transport` 使用 `nusb` 实现真实 USB 传输。

核心库不依赖命令行参数或终端输出。平台 USB 句柄也不会进入资产与协议的公共领域模型。

### `flypro-cli`

命令行程序负责：

- 将用户参数解析为核心库的类型；
- 选择内置资产或显式覆盖文件；
- 执行风险确认和输入文件检查；
- 打开指定烧录器并展示操作结果。

CLI 只能调用核心库提供的领域 API，不直接拼接端点号、命令块或参数镜像。

## 数据流

```text
内置或外部 FlyPro 发布资产
  -> 严格解析与关联校验
  -> 器件、算法、配置及封装信息
  -> 操作参数与项目数据
  -> 器件参数镜像
  -> 算法准备会话
  -> 操作会话
  -> 类型化命令、负载与响应
  -> Transport
  -> nusb 平台后端
  -> SP10/SP20 烧录器
```

## 依赖与安全原则

1. 高层模块依赖低层抽象，CLI 和平台后端不能反向进入资产或协议模型。
2. 每条 USB 事务显式声明 command OUT、payload OUT、response IN 或 completion IN 阶段。
3. 外部资产不能绕过结构、名称、关联关系和 CRC 校验。
4. 未确认的字段不赋予猜测性的业务名称，解析时保留原始字节。
5. I/O 失败、短传输、超时或取消会使当前传输对象不可继续使用。
6. 破坏性操作不自动重试，调用方必须处理“设备执行状态未知”的情况。

当前仅保留两个 workspace 成员。只有当某个模块形成独立发布、复用或隔离边界时，才考虑
拆分新的 crate。
